<objective>
Fix the Raptor Strike stack-lag reported in Phase 01 UAT Test 6: "When I use raptor strike with aspect of the eagle up it seems it takes a while for the stacks to update."

Root cause (diagnosed in .planning/debug/raptor-strike-aspect-eagle-stack-lag.md, CONFIRMED): With Aspect of the Eagle (186289) active, Raptor Strike fires UNIT_SPELLCAST_SUCCEEDED under the ranged-variant spell ID 265189 — NOT the melee talent ID 186270. The 265189 ID is absent from the CONSUMERS table, so ClassifySpellID returns nil, FindTrackedSpell finds nothing, and ApplySpell (the only instant predictive decrement) never runs. The stack is corrected only by the slow UNIT_AURA -> ScheduleAuraVerify -> SyncFromAura path, which defers >= 0.05s and stretches up to AURA_VERIFY_DELAY = 1.25s in combat — the observed lag. The user has CONFIRMED via in-game /etrace capture that the spell ID is 265189; it is a verified literal, not an unknown.

This plan adds 265189 to the CONSUMERS table so ClassifySpellID returns "consumer" for it, making ApplySpell decrement the stack instantly via the existing plain -1 consumer branch — identical to the melee Raptor Strike (186270). No other logic change is needed: 265189 is not Takedown, so it takes the plain decrement path (lines 704-709), and the consumer branch already handles a plain -1 correctly.

Purpose: Restore the instant predictive decrement for the Aspect-of-the-Eagle ranged Raptor Strike, eliminating the lag.
Output: One new entry in the CONSUMERS table plus a regression test in spec/tip_spec.lua.
</objective>
