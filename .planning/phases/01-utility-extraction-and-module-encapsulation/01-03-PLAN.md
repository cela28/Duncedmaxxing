<objective>
Fix the Kill Command stack-overshoot flicker reported in Phase 01 UAT Test 3: "when I press Kill command the stacks go to 3 for one instant before dropping back to 2."

Root cause (diagnosed in .planning/debug/kill-command-stack-overshoot.md): the generator branch of Tip:ApplySpell predicts the Kill Command grant as 3 when the Twin Fangs flag is set, else 2. Twin Fangs is a Takedown (consumer) modifier, NOT a Kill Command modifier. When Twin Fangs is talented, the generator path over-predicts +3 on a Kill Command cast whose real grant is 2 (Primal Surge). The display jumps to the predicted 3, then SyncFromAura reads the true aura (2 applications) and snaps it back to 2 — the visible flicker.

This plan decouples the Kill Command (generator) grant from hasTwinFangs, leaves Twin Fangs scoped to the Takedown consumer path (already correct at lines 699-703), and derives the generator grant from a new Primal-Surge-aware field (base 1, +1 with Primal Surge) via a HasPrimalSurge() helper modeled on the existing HasTwinFangs().

Purpose: Eliminate the predictive over-count so the display reaches the correct value directly with no flicker.
Output: Corrected Tip:ApplySpell generator branch, a HasPrimalSurge() helper plus Tip.hasPrimalSurge field wired into the same lifecycle events as hasTwinFangs, and regression tests in spec/tip_spec.lua.
</objective>
